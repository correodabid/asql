Why ASQL?

Because deterministic replay, explicit boundaries, and historical visibility should be normal database workflows, not custom infrastructure.

Reproduce any bug exactly

Same WAL input, same state output. Always. Replay production issues deterministically instead of guessing.

Query any point in time

Read data as it existed at any commit. No CDC pipelines, no event sourcing boilerplate. Just SELECT * FROM users AS OF LSN 42.

Isolate data boundaries

Domains give you schema-level multi-tenancy with explicit cross-domain transactions. No accidental data leaks.

Single binary, zero config

Go binary. Append-only WAL. Automatic snapshots. Run embedded or as a server. No JVM, no sidecar, no runtime dependencies.

Use your existing tools

ASQL exposes a pragmatic PostgreSQL-compatible subset over pgwire for documented `pgx`, `psql`, and mainstream tool flows. No proprietary driver lock-in.

Scale when you need it

Start local. Add replication. Form a Raft cluster. Same API, same semantics, no migration.

AI-native by default

Ships with its own MCP server. Claude Desktop, Cursor, and Zed can query your data, inspect row history, and time-travel to any commit — natively, without a custom integration.

Core Features

Everything you need from a modern database, plus the things others can't give you.

Time-Travel Queries

Read any historical state.
No extra infrastructure.

Every commit gets a monotonic LSN. Query past state with AS OF LSN, inspect row-level change history with FOR HISTORY, and trace entity evolution through versioned references.

  • Point-in-time reads by commit LSN
  • Row-level change audit trail
  • Entity version history across related tables
  • No CDC, no event store, no external system
SQL 
SELECT * FROM users AS OF LSN 42;

-- What changed and when?
SELECT * FROM invoices FOR HISTORY;
-- → __operation | __commit_lsn | row data

-- Current system position
SELECT current_lsn();

-- Entity version at a specific point
SELECT entity_version(
  'recipes', 'recipe_aggregate', 'recipe-1'
);
Domain Isolation

Explicit boundaries.
No accidental coupling.

Every transaction declares its scope. Domains are isolated data boundaries with independent schemas, constraints, and indexes. Cross-domain work is explicit and atomic.

  • Schema-level isolation per domain
  • Implicit domain creation on first use
  • Cross-domain atomic transactions
  • Cross-domain reads via IMPORT
SQL 
-- Each domain: own schema, own rules
BEGIN DOMAIN billing;
CREATE TABLE invoices (
  id INT PRIMARY KEY,
  amount FLOAT,
  status TEXT
);
COMMIT;

-- Atomic cross-domain work
BEGIN CROSS DOMAIN billing, inventory;
INSERT INTO invoices VALUES (2, 100.0, 'paid');
UPDATE stock SET qty = qty - 1 WHERE pid = 42;
COMMIT;
Deterministic Replay

Same input.
Same output. Always.

The WAL is the source of truth. Materialized state is derived. Replay the same log and you get bit-for-bit identical results — no wall-clock dependencies, no hidden nondeterminism.

  • Append-only WAL as canonical truth
  • Automatic snapshots every 500 mutations
  • Crash recovery: snapshot + WAL tail replay
  • Injectable time and entropy for full determinism
Architecture 
  [Mutation]
      │
      ▼
  [WAL Append] ─────── source of truth
      │
      ▼
  [Apply to State] ── derived, rebuildable
      │
      ├── periodic ──▶ [Snapshot]
      │
      ▼
  [Recovery]
      │
      └── Snapshot + WAL tail ──▶ identical state
Entity Versioning

Track aggregate versions
across tables.

Define entities that span multiple tables. When any table in an entity changes, its version increments automatically. Foreign keys can capture the exact version at insert time.

  • Automatic entity version tracking
  • Versioned foreign keys with auto-capture
  • JOINs resolve to captured version automatically
  • Full version history per entity
SQL 
-- Entity spans multiple tables
CREATE ENTITY recipe (
  ROOT ingredients,
  INCLUDES steps, quality_checks
);

-- Versioned FK captures exact version
CREATE TABLE process_orders (
  id TEXT PRIMARY KEY DEFAULT UUID_V7,
  recipe_id TEXT,
  recipe_version INT,
  VERSIONED FOREIGN KEY (recipe_id)
    REFERENCES master.ingredients(id)
    AS OF recipe_version
);

Start simple. Scale when ready.

Same engine, same API. Choose your deployment model.

1

Local / Embedded

Single binary. Zero config. Perfect for development, testing, edge, and small workloads.

go run ./cmd/asqld \
  -addr :5433 \
  -data-dir .asql
2

Server (pgwire)

Run as a standalone server. Connect with any PostgreSQL-compatible client or driver.

conn, _ := pgx.Connect(ctx,
  "postgres://localhost:5433/asql")

conn.Exec(ctx, "BEGIN DOMAIN billing")
conn.Exec(ctx, `UPDATE invoices SET
  status = 'paid' WHERE id = 42`)
conn.Exec(ctx, "COMMIT")

// Same driver. Same SQL. Plus history:
conn.Query(ctx, `SELECT * FROM invoices
  AS OF LSN 100`)
3

Cluster (Raft)

Add nodes for replication and high availability. WAL streaming, read routing, automatic catch-up.

go run ./cmd/asqld \
  -addr :5433 -data-dir .node-a \
  -node-id node-a -grpc-addr :6433 \
  -peers node-b@:6434,node-c@:6435 \
  -groups default

Up and running in 2 minutes

1

Start the engine

$ go run ./cmd/asqld -addr :5433 -data-dir .asql
2

Open the interactive shell

$ go run ./cmd/asqlctl -command shell -pgwire 127.0.0.1:5433
3

Create data and travel through time

BEGIN DOMAIN app;
CREATE TABLE users (id INT PRIMARY KEY, name TEXT, status TEXT);
INSERT INTO users VALUES (1, 'Alice', 'active');
COMMIT;

-- Check current position
SELECT current_lsn();  -- → 3

-- Make a change
BEGIN DOMAIN app;
UPDATE users SET status = 'suspended' WHERE id = 1;
COMMIT;

-- Travel back in time
SELECT * FROM users AS OF LSN 3;
-- → Alice is still 'active'

-- Full audit trail
SELECT * FROM users FOR HISTORY;
-- → INSERT at LSN 3, UPDATE at LSN 4

Familiar SQL. Unique capabilities.

Standard SQL where it matters. Unique primitives where they count.

DDL

CREATE TABLE products (
  id TEXT PRIMARY KEY DEFAULT UUID_V7,
  name TEXT NOT NULL,
  price FLOAT CHECK (price > 0),
  category_id INT REFERENCES categories(id)
);

CREATE INDEX idx_name
  ON products USING btree (name);

Queries

SELECT c.name, COUNT(*), AVG(p.price)
FROM products p
JOIN categories c ON p.cat_id = c.id
GROUP BY c.name
HAVING COUNT(*) > 5;

WITH top AS (
  SELECT *, RANK() OVER (
    ORDER BY price DESC)
  FROM products
)
SELECT * FROM top WHERE rank <= 10;

Transactions

BEGIN DOMAIN billing;

SAVEPOINT before_charge;
UPDATE accounts
  SET balance = balance - 100
  WHERE id = 1;

-- Changed your mind?
ROLLBACK TO SAVEPOINT before_charge;

COMMIT;

Cross-domain reads

BEGIN DOMAIN analytics;

-- Read from another domain
IMPORT billing.invoices AS inv;

SELECT u.name, SUM(inv.amount)
FROM users u
JOIN inv ON u.id = inv.user_id
GROUP BY u.name;

COMMIT;
