# ASQL GA Compatibility Contract v1

Date: 2026-03-15
Status: draft contract for Epic P closure and GA release gating.

## Purpose

This document defines the compatibility promises ASQL intends to freeze for GA.

It exists to answer four questions clearly:

1. what ASQL guarantees to keep stable across GA releases,
2. what remains explicitly outside the contract,
3. what evidence is required before a release can claim the contract still holds,
4. how contract changes must be documented and versioned.

## Contract stance

ASQL is a deterministic SQL engine with a pragmatic PostgreSQL-compatible subset over pgwire.

GA compatibility means:
- the documented ASQL-native transaction and temporal model remains stable,
- the documented pgwire compatibility subset remains stable,
- WAL and recovery behavior remain stable enough for supported upgrade and restore flows,
- contract changes are explicit, versioned, and release-noted.

GA compatibility does **not** mean:
- drop-in PostgreSQL parity,
- permanent stability for undocumented behavior,
- support for direct use of internal files, snapshots, or hidden admin/debug surfaces outside documented workflows.

## Compatibility domains

The GA contract is divided into five domains.

### 1. SQL and pgwire application contract

ASQL guarantees stability for the documented canonical runtime path:
- `cmd/asqld` as the primary server runtime,
- pgwire as the main application-facing interface,
- the documented SQL subset used by getting-started, fixtures, SDK examples, and mainstream supported client/tool flows.

This includes, when documented and test-backed:
- pgwire startup and authentication behavior within the supported subset,
- simple query flow and supported extended-query flow,
- ASQL-native transaction primitives:
  - `BEGIN DOMAIN <name>`
  - `BEGIN CROSS DOMAIN <a>, <b>, ...`
  - `COMMIT`
  - `ROLLBACK`
- supported DDL/DML/query shapes described in current docs,
- temporal query surface such as `AS OF LSN`, `FOR HISTORY`, and documented helper functions,
- supported SQLSTATE/error-shape expectations documented for compatibility-critical flows.

The authoritative surface definition remains:
- [sql-pgwire-compatibility-policy-v1.md](sql-pgwire-compatibility-policy-v1.md)
- [postgres-compatibility-surface-v1.md](postgres-compatibility-surface-v1.md)

### 2. WAL and replay contract

ASQL guarantees that supported releases can:
- read and replay WAL generated by the immediately supported upgrade window,
- reject incompatible WAL explicitly rather than silently misinterpreting it,
- preserve deterministic replay semantics for the documented engine surface.

At minimum, the GA contract requires:
- WAL versioning to remain explicit,
- version mismatch handling to remain explicit and test-backed,
- append sequence continuity checks to remain in the release gate,
- replay equivalence and restart parity to remain release-gated.

This contract applies to documented engine/runtime upgrade flows, not to arbitrary manual editing of WAL artifacts.

### 3. Backup, restore, and upgrade contract

ASQL guarantees that the documented backup/restore and upgrade workflows remain supported and test-backed.

That means a supported release must preserve the documented ability to:
- restart from its WAL cleanly,
- restore from supported backups,
- recover historical state through the documented restore/time-travel flows,
- upgrade from supported predecessor versions inside the documented upgrade window.

If the supported upgrade window changes, that change must be documented explicitly in release notes and compatibility docs.

### 4. Operator contract

ASQL guarantees stability for the documented operator-facing surfaces needed to run the canonical product:
- health and readiness endpoints,
- metrics required by current dashboards/SLO docs,
- documented leadership/failover inspection endpoints,
- documented admin/recovery inspection workflows.

The operator contract is about shape and availability of documented workflows, not every internal metric label or temporary debug field.

### 5. Documentation and claim contract

A behavior is part of the GA contract only when all of the following are true:
- it is documented in the relevant public doc,
- it has release-quality regression evidence,
- it is not marked experimental, partial, legacy, or internal-only.

Undocumented behavior is not part of the contract.

## Explicit non-contract areas

The following are outside the GA compatibility contract unless later documented otherwise:
- full PostgreSQL parity,
- undocumented catalog quirks or tool-specific side effects,
- internal file formats beyond the documented WAL/backup compatibility window,
- hidden or ad hoc admin/debug endpoints,
- legacy/non-canonical runtime paths not presented as the supported production path,
- performance of arbitrary unbenchmarked workloads.

## Change policy

### Allowed in a patch/minor release
- internal refactors that preserve documented behavior,
- additional compatibility/test coverage that does not narrow support,
- new optional behavior documented as additive and test-backed,
- improved diagnostics or SQLSTATE fidelity that tightens the documented subset without breaking existing documented flows.

### Not allowed without explicit contract/version change
- narrowing a documented supported pgwire/client/tool flow,
- changing WAL interpretation for a supported upgrade path without versioned migration handling,
- breaking documented backup/restore or replay semantics,
- changing documented temporal/query/operator behavior silently.

### If a contract change is unavoidable
- version the contract document,
- update the compatibility matrix/evidence docs in the same window,
- record the change in release notes and upgrade guidance,
- provide an explicit migration or rollback story where applicable.

## Required release evidence

A release may claim GA contract compliance only if the following are green in the same release window.

### Application / compatibility evidence
- baseline lanes from [../operations/pgwire-compatibility-test-pack-v1.md](../operations/pgwire-compatibility-test-pack-v1.md)
- compatibility docs reviewed and aligned
- focused SQLSTATE/error-shape regressions when startup/auth/protocol/error behavior changed

### WAL / replay evidence
- WAL compatibility/version tests green
- replay/restart parity green
- deterministic recovery evidence recorded in the release bundle

### Backup / restore / upgrade evidence
- backup/wipe/restore parity green
- restore boundary and corruption detection tests green
- supported upgrade simulation executed and recorded

### Operator evidence
- health/readiness/metrics/admin sanity green
- documented operator endpoints and metrics still align with operations docs

### Cluster continuity evidence
Required whenever cluster/runtime, failover, fencing, or replicated commit behavior changed:
- failover continuity suite green
- replay state-hash continuity green

## Relationship to performance guardrails

Performance is not part of API compatibility, but it is part of release confidence.

For GA, the following should be treated as mandatory release evidence alongside compatibility:
- single-node write scaling guardrail,
- cluster append-growth guardrail,
- benchmark baselines for the canonical write/read/recovery paths.

This prevents "compatible but operationally regressed" releases from passing the gate.

## Bottom line

For GA, ASQL should promise stability only where it has all three:
- explicit documentation,
- repeatable regression evidence,
- a clear operator and upgrade story.

Anything else should remain outside the contract until it is documented and proven.
